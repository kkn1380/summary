# 정적 vs 동적 페이지 비교

## 📊 구조 비교

### 기존 방식 (main 브랜치)
```
매번 monitor 실행 시:
┌─────────────────────────────────────┐
│ 1. 새 영상 발견                      │
│ 2. 자막 추출 + AI 요약               │
│ 3. latest.json 생성 (데이터)         │
│ 4. index.html 재생성 (데이터 embed)  │ ← 매번 변경
│ 5. index.mobile.html 재생성          │ ← 매번 변경
└─────────────────────────────────────┘

파일 크기:
- latest.json: ~50KB
- index.html: ~150KB (데이터 포함)
- index.mobile.html: ~150KB (데이터 포함)
총: ~350KB
```

### 새 방식 (feature/dynamic-page 브랜치)
```
최초 1회:
┌─────────────────────────────────────┐
│ 1. index.html 생성 (템플릿만)        │ ← 한 번만
│ 2. index.mobile.html 생성            │ ← 한 번만
└─────────────────────────────────────┘

매번 monitor 실행 시:
┌─────────────────────────────────────┐
│ 1. 새 영상 발견                      │
│ 2. 자막 추출 + AI 요약               │
│ 3. latest.json만 업데이트            │ ← JSON만 변경
└─────────────────────────────────────┘

파일 크기:
- latest.json: ~50KB (매번 변경)
- index.html: ~30KB (고정)
- index.mobile.html: ~30KB (고정)
총: ~110KB
```

## 🎯 핵심 차이점

| 항목 | 기존 (정적) | 새 방식 (동적) |
|------|------------|---------------|
| **HTML 생성** | 매번 | 최초 1회만 |
| **데이터 위치** | HTML 내부 embed | 별도 JSON 파일 |
| **Git diff** | HTML 전체 변경 | JSON만 변경 |
| **파일 크기** | 큼 (~350KB) | 작음 (~110KB) |
| **로딩 방식** | 즉시 렌더링 | fetch 후 렌더링 |
| **서버 필요** | ❌ | ❌ |
| **SEO** | ✅ 완벽 | ⚠️ CSR (약간 불리) |
| **브라우저 호환** | ✅ 모든 브라우저 | ✅ 최신 브라우저 (fetch API) |

## 💡 장단점

### 기존 방식 장점
- ✅ SEO 최적화 (데이터가 HTML에 포함)
- ✅ 즉시 렌더링 (fetch 불필요)
- ✅ 오래된 브라우저 지원

### 기존 방식 단점
- ❌ 매번 HTML 재생성 (비효율)
- ❌ Git diff가 복잡 (HTML 전체 변경)
- ❌ 파일 크기 큼
- ❌ HTML 템플릿 수정 시 재배포 필요

### 새 방식 장점
- ✅ 효율적 (JSON만 업데이트)
- ✅ Git diff 간단 (JSON만 변경)
- ✅ 파일 크기 작음
- ✅ HTML 템플릿과 데이터 분리
- ✅ 캐싱 전략 적용 가능
- ✅ 여전히 서버 불필요

### 새 방식 단점
- ⚠️ SEO 약간 불리 (CSR)
- ⚠️ 초기 로딩 시 fetch 필요 (매우 빠름)
- ⚠️ 오래된 브라우저 미지원 (IE11 등)

## 🔍 실제 예시

### Git Diff 비교

#### 기존 방식
```diff
# 새 영상 1개 추가 시
 data/site/latest.json        | 50줄 변경
 data/site/index.html          | 3000줄 변경 ← 전체 재생성
 data/site/index.mobile.html   | 3000줄 변경 ← 전체 재생성
```

#### 새 방식
```diff
# 새 영상 1개 추가 시
 data/site/latest.json        | 10줄 변경 ← JSON만 변경
```

## 🚀 성능 비교

### 초기 로딩
```
기존 방식:
1. index.html 다운로드 (150KB)
2. 즉시 렌더링
총: ~150KB, ~100ms

새 방식:
1. index.html 다운로드 (30KB)
2. latest.json fetch (50KB)
3. 렌더링
총: ~80KB, ~150ms
```

### 업데이트 후 재방문
```
기존 방식:
1. index.html 다시 다운로드 (150KB) ← 전체 변경
총: ~150KB

새 방식:
1. index.html 캐시 사용 (0KB) ← 변경 없음
2. latest.json만 다운로드 (50KB)
총: ~50KB ← 3배 빠름!
```

## 🎨 프레임워크 필요성

### 질문: 프레임워크가 필요한가?

**답: 아니요! 순수 JavaScript로 충분합니다.**

#### 현재 프로젝트 규모
- 단순한 리스트 렌더링
- 검색 기능
- TTS 기능
- 날짜별 그룹핑

#### 프레임워크가 필요한 경우
- 복잡한 상태 관리
- 많은 컴포넌트 재사용
- 실시간 업데이트
- 복잡한 라우팅

#### 프레임워크 비교

| 프레임워크 | 번들 크기 | 빌드 필요 | 학습 곡선 | 현재 프로젝트 적합성 |
|-----------|----------|----------|----------|-------------------|
| 순수 JS | 0KB | ❌ | 낮음 | ✅ 완벽 |
| React | ~40KB | ✅ | 중간 | ❌ 오버킬 |
| Vue | ~30KB | ✅ | 낮음 | ❌ 불필요 |
| Svelte | ~2KB | ✅ | 중간 | ❌ 불필요 |

## 🖥️ 서버 필요성

### 질문: 서버가 필요한가?

**답: 아니요! 정적 호스팅으로 충분합니다.**

#### 정적 호스팅 (추천 ✅)
```
GitHub Pages, Netlify, Vercel 등
- 비용: 무료
- 설정: 간단
- 성능: 빠름 (CDN)
- 유지보수: 불필요
```

#### 서버 사이드 (불필요 ❌)
```
Express, Next.js 등
- 비용: 유료 (서버 운영)
- 설정: 복잡
- 성능: 서버 위치에 따라 다름
- 유지보수: 필요
```

#### 서버가 필요한 경우
- 데이터베이스 연동
- 사용자 인증
- 실시간 업데이트 (WebSocket)
- 서버 사이드 로직
- API 프록시

**현재 프로젝트는 해당 없음!**

## 🚀 마이그레이션

```bash
# 1. 새 브랜치로 전환
git checkout feature/dynamic-page

# 2. 빌드
npm run build

# 3. 기존 HTML 삭제 (새로 생성하기 위해)
rm data/site/index.html data/site/index.mobile.html

# 4. 실행
npm run monitor

# 5. 로컬 테스트
npm run serve:site
# 브라우저에서 http://localhost:8000 접속

# 6. 만족하면 main에 병합
git checkout main
git merge feature/dynamic-page
```
